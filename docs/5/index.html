5. Making our first game

We're going to work on a game that has more interactivity. It's a bit like Pressy Pressy meets Tetris, called Pile Up.

To do this you should open main.py and find the line that says engine.loadGame(games.PressyPressy()). This is the code that tells the game engine which game we want to run when the Plasma is switched on. You can edit this line to load different games and animations. Further down the file there is a list of the games and animations that come pre installed. Let's try a cool animation.

Edit the line so it now says: engine.loadGame(animations.FireEffect()). Click Play.

These animations are not interactive (apart from Demo), so are more like traditional animations for fairy lights. You can leave it like this attached to your tent for a more traditional fairy light vibe (i.e. you want your tent to look like it's on fire). Try animations.AlternatingBlinkies() for a more chill one. Feel free to experiment with these later.

For now, let's load the PileUp game. Change the line to be engine.loadGame(games.PileUp())

Press Play.

The yellow pixel (the player) moves slowly down the LED strip. When it gets to the bottom it starts again from the top. You might want to lay out your fairy lights so you can see it more clearly. It looks even better suspended from the ceiling or blue tacked to a wall!

Press the button. It should turn the pixel the player was on red, and then reset the player to the top again. The aim of the game is to turn all the pixels red.

It works but it's a bit boring so let's add a way for the player to lose the game. We'll change it so that if the player touches a red light, they lose and the game restarts.

Game Files

In Thonny, at the bottom left, open the folder on the plasma called games and open the file called pileup.py. This is the source code for our "game cartridge". All the other games and animations have similar structures.

The code here has loads of comments to help you understand what is happening in this game. Broadly there are three key "methods" (blocks of code) that every game cartridge has:

Init

This is code that is run when the game is loaded for the first time, it is where we set up any variables or other code we need for our specific game.

In PileUp it makes a "game board" that we use to track where the red pixels are, and sets the player position.

update

This code is run by the game engine about 60 times per second, and is where we handle the logic, or the rules of the game.

In PileUp, this moves the player down one pixel, and also detects if the button has been pressed. If it has been, it changes the game board to keep track of that.

draw

This code is where we actually change the pixels on the LED strip.

In PileUp, it cycles through the game board, making the pixels red if there is something there, black/off if there isn't anything. Finally it lights up the pixel where the player is.

Adding Danger 💥

We want the player to lose if they hit a red light. This is a logical problem: we need to check if the player shares a position on the game board. Then, if they do, make them lose. That means we need to edit the update method. Look through that section of code and find the line that checks if there is a collision.

if (self.board[self.position] != 0):
    pass

If you have not done much programming, this does a check: If the part of the game board, at the same position as the player, is not equal to zero, then run the next line of code (pass)

However pass is just a placeholder - it doesn't do anything. Delete pass and make it match the following code:

if (self.board[self.position] != 0): 
    gameover(engine)
    self.init(engine)

Remember that Python is strict about indentation. Make sure the code is lined up properly. Especially, the two lines underneath the if should be aligned further right than the if (see above). This tells python to only run that code if the condition is met. Otherwise the player would always be losing all the time!

This code does two things. First it runs a little "game over" animation that flashes the LED strip red on and off a few times. Then, self.init(engine) basically reloads the cartridge, which sets the game back to the start again.

Running the game

If you try and click play here, you will probably get an error. This is because we can't run this code directly, we have to run it through the main code. So open main.py in Thonny again (it probably is open in a tab), and play from there. It will load your cartridge and run it.

Try and see how many lights you can manage to turn red before losing. Timing is everything, it will take a bit of practice!

Tweaks

Great work! Before we move on to the next game, let's have a quick look at some other bits of the game that are good to understand:

Changing the Difficulty

There's a line at the bottom of the draw method that says time.sleep(0.1). This forces the Plasma to wait for 0.1 seconds. This just makes the game a bit slower. Try changing this value to make the game faster (try 0.0, or 0.05) or slower (try 0.5 or 1.0). The slower speeds make the button laggy - you have to be holding the button when the sleep ends.

Changing the Colours

Right near the bottom of the file there is a line that looks like this:

display.set_rgb(self.position, 120, 120, 0)

This code tells the Fairy lights to set the colour of a specific fairy light (the one at self.position - the location of the player) and gives three numbers: 120,120,0.

Each fairy light is actually three LEDs - one red, one green, and one blue. Just like pixels on televisions, computer monitors and phone screens. When we mix these colours together in different brightnesses, it gives the impression of more complex colours. The value for each colour is between 0 and 255, where 0 is off, and 255 is maximum brightness. So our current code says that the pixel where the player is should have the red brightness set to 120, the green brightness to 120, and the blue brightness to 0 (i.e. off). When we mix red and green in equal quantities in additive colour (light) we get yellow. For more background on RGB check out this wikipedia article.

Have a go at changing the colour of the player. You can either just experiment by changing these three values to any numbers between 0 and 255, or by using a tool like this online RGB colour picker to pick an interesting colour and then change the R, G, and B numbers to match.

Some things to note are that this is not an exact science - some colours don't come out as well as others. For example, we don't really have a "black" because our leds are encased in a white blob. So dark colours just look faint.

You should also avoid using maximum brightness, and especially maximum white (255,255,255). For two reasons: first, it is a bit too bright! Especially at night, one LED at max brightness will keep you awake. Secondly, the higher the brightness the more power you are using. If you turn all the LEDs on maximum brightness white all at once, you might damage the Plasma, or just drain your power bank really quickly.

I strongly recommend just thinking of 128 as the maximum, instead of 255. This gives plenty of wiggle room, and is still bright enough (look at the yellow that 120,120,0 made - it is fine).

Extra

We talked about RGB above, but when programming LEDs, you often want to do calculations on colour, and RGB can be awkward for that. Instead we use HSV (Hue, Saturation, Value). RGB makes more sense in our brains, but HSV makes it easier to transition between colours logically. Each of H,S, and V is a value between 0 and 1.0. Changing H changes the colour in a circular way (0.0: red, 0.3: green, 0.5: cyan, 0.7: purple, 1.0: red again), changing S changes the saturation (0.0: grey, 1.0: maximum vividness) and V changes brightness (0: off, 1.0: full brightness)

For example of why this is handy, as mentioned above, it's best to keep brightness to about half at maximum. In RGB that means changing every value, but in HSV, we only need to change the "V".

Replace the following line:

display.set_rgb(self.position, 120, 120, 0)

With the equivalent in HSV:

display.set_hsv(self.position, 0.17, 1.0, 0.6)

As before you can use the online colour picker, just look at the Hue and Saturation values. The "V" can be left as 0.6 - the brightness is now independent of the colour.

HSV is particularly great for one thing we love: rainbows 🌈 Because the hue value is just between 0.0 and 1.0, we can cycle through it and make things colourful easily. Try changing the line to this:

display.set_hsv(self.position, (self.position/50.0), 1.0, 0.6)

Now the colour of the player changes based on how far along the LED strip it is (so, its position divided by the length of the strip: 50 in this case, gives a number between 0 and 1.0, which we use as the hue value). The saturation and value stay the same for all colours.

Advanced

The walls are always red, but the way we set the colour looks like this:

display.set_hsv(i, self.board[i], 1.0, 0.6)

So the hue is actually the value of the position in the game board. If you figure out where that is set, you can store any value between 0.0 and 1.0 and have the board use variable colours too. maybe based on position, but also many based on time? so the game board can turn more red as more walls are built?

Careful: the game can get very confusing if it is a multicolour mayhem!

Also, look in utils/gameover.py - this is the mini animation that gets played when the player loses. Have a go editing this and making your own game over scene. Maybe a cylon/kitt style wipe? It is shared between the next few games, so any changes there will also affect the next examples.
